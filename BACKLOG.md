## Issue #6: RAG indexer + build script

**Labels:** `agent-ready`, `tdd`

### What to build

Implement `lib/rag` with the stable interface `{ build(docs): Promise<Chunk[]>, query(question, k): Promise<Chunk[]> }`. Build chunks text into ~800-char windows with overlap, embeds each chunk with Gemini `text-embedding-004`, persists the index as JSON. Query embeds the question, computes cosine similarity vs. all chunks, returns top-k with their metadata. Implement `scripts/rag-build` as a Node CLI that reads `content/resume.json` + `content/projects/*.mdx` + `content/posts/*.mdx`, calls `lib/rag.build`, and writes `data/rag-index.json`. Wire `scripts/rag-build` into `npm run build` via a `prebuild` script. Both `lib/rag` and the build script get TDD with fixtures.

### Acceptance criteria

- [ ] `lib/rag.ts` exports the interface described above
- [ ] Embedding + chat calls are wrapped behind an injectable interface; tests use a stub embedder
- [ ] Vitest covers: chunking produces expected number of chunks with overlap; `query` returns chunks ordered by cosine similarity; round-trip (build → write → read → query) returns identical results
- [ ] `scripts/rag-build` reads from `content/` and writes a valid `data/rag-index.json`
- [ ] Vitest covers the script with a fixture content directory
- [ ] `prebuild` script in `package.json` runs `rag-build` before `next build`
- [ ] Running `npm run build` regenerates the index from the latest content (no stale embeddings)
- [ ] Build fails loud if `GEMINI_API_KEY` is missing
- [ ] The index file is < 5 MB at launch (sanity check on bundle size)

## Issue #14: Rate-limit fallback + anonymous chat logs

**Labels:** `hitl`

### What to build

If `lib/chat.ask` catches a Gemini rate-limit or server error, fall back to a static-FAQ mode: a `content/faq.json` file with the top 10 expected questions and pre-written answers; `lib/chat` performs a simple text-similarity match against FAQ entries and returns the matched answer (or a "this isn't a question I have a canned answer for; please email Abhishek" if no match). Separately, log every question sent to `/api/chat` anonymously (no IPs, no user agents, no fingerprints) so the owner can later see what recruiters care about. Storage approach (Vercel KV vs. simple log to a file via a serverless function) is a decision in this slice.

### Acceptance criteria

- [ ] `content/faq.json` exists with ≥ 10 question/answer pairs
- [ ] Simulating a Gemini rate-limit (mock) routes the request to the FAQ matcher
- [ ] FAQ matcher returns the matched answer or the "no canned answer" template
- [ ] Each `/api/chat` call writes one log entry with: timestamp, question, retrieved-chunk-source slugs, fallback-used boolean
- [ ] Logs contain no IP, no user-agent, no cookie data, no user identifier
- [ ] Owner decides storage approach: `[ ] Owner: chose storage (Vercel KV / file / other)`
- [ ] An owner-facing way to read recent logs exists (a CLI script, a static endpoint behind a token, or Vercel KV CLI — owner picks)
