## Problem Statement

I'm a Computer Science undergraduate (3rd year) at UIT-RGPV Bhopal, currently working as a Full-Stack Developer at SimptionTech where I built a production multi-tenant SaaS (Simption ERP). I want to land Software Development Engineer roles — internships and full-time — with a clear lean toward foreign (US/EU) companies.

My current public footprint (GitHub + LinkedIn + resume PDF) doesn't communicate the rare combination I actually bring: **AI-native product engineering on real production foundations**. Recruiters skim hundreds of generic dev portfolios per week. Without a portfolio that proves the thesis in the first 10 seconds, I look like every other CS undergrad with toy projects.

The constraints that make this hard:

- I'm early-career, so I can't out-credential a senior engineer — I have to out-execute on signal-to-noise.
- "AI projects" is a saturated claim; most candidates ship wrapper apps, not real engineering.
- A standard dev portfolio (dark mode, project cards, gradient hero) blends straight into the recruiter's queue.

## Solution

Build a personal portfolio that:

- Reads as **credibly senior** in the first 5 seconds (Linear-grade dark minimal aesthetic, no AI-slop tropes).
- Proves the **"AI-native + production engineering"** thesis automatically via a working **"Ask my portfolio"** RAG agent embedded in the hero.
- Demonstrates **system thinking** via architecture diagrams as the primary visual on every project card (not UI screenshots).
- Backs the AI thesis with **3–4 deep technical writeups** that recruiters can use to verify the candidate's depth.

The signature distinctiveness comes from two design choices no competing portfolio has:

1. **A working AI agent** in the hero, indexed over the candidate's own resume + project descriptions, that recruiters can interact with directly.
2. **Architecture diagrams** (multi-tenant flow, RAG pipeline, OpenVINO model chain) as the dominant project visual — instead of UI screenshots.

The base aesthetic is conservative (dark, restrained, Geist/Inter sans, sharp grid, single accent color). That conservatism is what makes the two signature elements feel intentional rather than gimmicky.

## Implementation Decisions

### Stack

- **Framework**: Next.js 15 (App Router) + TypeScript (strict).
- **Styling**: Tailwind CSS + CSS variables for design tokens generated by `ui-ux-pro-max`. shadcn/ui for primitives (Button, Dialog, Tooltip, ScrollArea).
- **Motion**: `framer-motion` (already installed).
- **Content**: MDX for `/writing` posts and project case studies. Processed by `next-mdx-remote` (avoids the build-time complexity of `@next/mdx`).
- **AI**: Gemini API (matches the candidate's existing stack and brand consistency).
- **Persistence**: No database. The RAG index is a static JSON file generated at build time and shipped with the bundle. Chat is stateless per request.
- **Deployment**: Vercel.
- **Domain**: pending owner decision; default to a `*.vercel.app` subdomain until provided.

### Modules

**Deep / TDD'd:**

1. **`lib/rag`** — RAG indexer + retriever
   - Inputs: array of text documents with metadata (source, type, project slug).
   - Pipeline: chunk text → embed via Gemini `text-embedding-004` → store as JSON.
   - Query: embed the user question → cosine similarity vs. all chunks → top-k.
   - Stable interface:
     ```ts
     type Doc = { id: string; source: string; text: string; meta: Record<string, unknown> };
     type Chunk = { docId: string; text: string; embedding: number[]; meta: Record<string, unknown> };
     interface RagIndex {
       build(docs: Doc[]): Promise<Chunk[]>;
       query(question: string, k: number): Promise<Chunk[]>;
     }
     ```
   - Persistence is internal — `build` writes the JSON, `query` reads it lazily.

2. **`lib/chat`** — Chat agent service
   - Interface: `ask({ question, history }): AsyncIterable<string>` (yields token chunks).
   - Composes the system prompt: *"You answer questions about Abhishek Choudhary's work. Only use the provided context. If a question is unrelated to his work, decline politely. Cite sources by project slug."*
   - Calls `lib/rag.query(question, 6)`, formats context, calls Gemini chat, streams the response.
   - Refuses unrelated questions via system prompt + a lightweight intent check (heuristic: if no context chunk above similarity threshold, reply with a fixed refusal template).
   - All Gemini calls happen server-side; the API key never reaches the client.

3. **`scripts/rag-build`** — build-time indexer
   - Reads `content/projects/*.mdx`, `content/posts/*.mdx`, `content/resume.json`.
   - Calls `lib/rag.build(docs)` → writes `data/rag-index.json`.
   - Wired into `npm run build` via the `prebuild` script.

4. **`content/schema`** — Zod schemas for `Project` and `Post` MDX frontmatter.
   - Validated at build time; failed validation fails the build (loud).

**Shallow / not TDD'd:**

5. **`components/diagram/`** — `<ArchitectureDiagram>` (inline SVG per hero project), `<MermaidBlock>` (client-side render for posts).
6. **`components/motion/`** — `<FadeUpOnScroll>`, `<MagneticButton>`, `<ScrambleText>` (subtle hero accent).
7. **`components/chat/`** — `<AskMyPortfolio>` UI: input, streamed answer area, source citation list, loading shimmer.
8. **`components/layout/`** — `<Header>` (sticky, backdrop-blur, current-section highlight), `<Footer>`, page-transition wrapper.
9. **`components/project/`** — `<ProjectCard>`, `<ProjectCaseStudy>`.
10. **`app/api/chat/route.ts`** — server-side handler (Edge runtime). Calls `lib/chat.ask`, returns an SSE stream.
11. **`app/`** — routes: `/`, `/work`, `/work/[slug]`, `/writing`, `/writing/[slug]`, `/about`, `/contact`.

## Testing Decisions

**What makes a good test here**:

- Test external behavior, not implementation details. Don't assert "the LLM was called with this exact string" — assert "given these docs + this question, we got chunks back with the expected docIds."
- Use deterministic fixtures. Real Gemini calls don't go in tests — embed/chat calls are wrapped behind interfaces and we inject fakes for unit tests.
- One integration test per critical flow uses the real (or recorded) API, gated behind an environment flag so CI doesn't pay for it on every run.

**Modules tested**:

- `lib/rag`:
  - Given a fixed set of docs and a stub embedder returning known vectors, `query` returns chunks ordered by similarity.
  - Chunking strategy: a 5000-character doc produces N chunks of expected approximate size with overlap.
  - Persistence round-trip: `build` → write → re-read → `query` yields identical results.
- `lib/chat`:
  - Given a mocked RAG returning 0 chunks (off-topic), the agent emits the refusal template.
  - Given mocked RAG returning chunks, the system prompt includes those chunks in order with source labels.
  - Streaming: the iterable yields > 0 tokens and terminates cleanly.
- `scripts/rag-build`:
  - Given a fixtures directory with 2 projects + 1 post, the script writes a JSON file with the expected number of chunks.
- `content/schema`:
  - Valid frontmatter passes; missing required fields throws with a clear error.

**Not tested (visual + composition)**:

- Components, pages, motion. Verified manually with the `run` skill on every meaningful change, plus the `verify` skill before launch.

**Prior art**:

- No existing tests in this repo. Test runner: **Vitest** (already listed on the candidate's mini-resume stack). Tests colocated as `*.test.ts` next to the modules they cover.
